The Production Era Begins

A Clean Slate (That Needed More Cleaning)

The Digiquarium has been through a lot. A beta period where we discovered we were running the wrong prompt version. A massive v2.0 cleanup that consolidated the entire codebase. And then, when we tried to actually run the system in production on March 27, we discovered that the cleanup hadn't gone deep enough.

The original version of this post was written with optimism: Ollama is local, baselines are fresh, we're good to go. The reality was that it took another 36 hours of intensive work to get the system to a state where that optimism was justified.

Ollama Comes Home — For Real

The biggest infrastructure change: Ollama is no longer running on a separate Mac Mini and accessed over the local network through a socat proxy container. It's now running inside Docker on the same host as everything else. This eliminates the network dependency that caused intermittent connection timeouts during the beta period, and it means the entire stack — tanks, daemons, Ollama, Kiwix — lives in one docker-compose.yml.

But just moving Ollama local wasn't enough. With 17 tanks all hitting one model simultaneously, we needed concurrency control. The solution was threefold: a file-based mutex lock so only one tank talks to Ollama at a time, a deterministic stagger that spreads requests across a 170-second window by tank ID, and a rotation daemon that groups tanks into sets of 3 with 5-minute windows. Ollama also has auto-restart via supervisor and health checks built into the scheduler.

Custom Tank Image

Tanks were crash-looping because pip install ran at startup inside each container. If the network was slow or a package was unavailable, the tank would fail before it even started exploring. We built a custom Docker image with all Python dependencies pre-installed. Tanks now start in seconds instead of minutes, and they never fail due to package installation.

Fresh Baselines — Done Right

Every single tank has been baselined from scratch. All 17 specimens started their production runs with a clean 14-question personality assessment. But this time, baselines run sequentially (not all 17 at once), with Ollama health checks between each run, and the output files include timestamps so they never overwrite previous data.

✅ Infrastructure Verified

All 17 YAML configs validated. Static IPs assigned. Ollama running locally with mutex. Custom tank image deployed. Sequential baseline runner operational. All 21 daemons running with continuous loops and SLA tracking.

21 Daemons, Actually Working

The daemon audit revealed that 5 daemons had no continuous run loop — they'd execute their check once and exit. Another 11 had duplicate process issues from missing singleton locks. All 21 daemons were fixed to run properly: continuous loops with sleep intervals, SLA tracking (cycle count, success rate, average cycle time), and proper error handling.

The scheduler was completely rewritten to v4 with clash prevention (no two tanks scheduled simultaneously), Ollama health checks before dispatching, and overnight safety modes.

Explorer Actually Explores Now

The explorer was fundamentally broken for Kiwix navigation. It assumed standard Wikipedia URL formats, but Kiwix serves articles at different paths with different link structures. The fix added KIWIX_URL and WIKI_BASE environment variables, rewrote link extraction to handle Kiwix's format, and fixed random article selection. Without this fix, tanks were sitting idle instead of exploring.

Security Hardened

A security audit found a hardcoded password in the admin panel and an XSS vulnerability in visitor-facing pages. The password was removed (auth restructured), and input sanitization was added for all user-facing inputs.

Everything Else

Admin panel built at /admin/ with container status, daemon SLA metrics, and system health. Translation layer rewritten with actual translation logic. Brain minimap zoom bug fixed with cumulative chips displaying correctly. Mobile responsive hamburger navigation added to all pages. All nav links converted to absolute paths from site root.

MCP Server Connected

The Digiquarium now has an MCP server that allows external AI agents to inspect and operate the aquarium autonomously. Container status, health checks, log access, configuration, shell execution — it's all available through the MCP interface. This is what makes truly autonomous operation possible.

What Comes Next

With clean infrastructure, verified baselines, and autonomous tooling in place, the focus shifts to what the Digiquarium was always about: watching what happens when AI specimens explore knowledge freely over extended periods. The plumbing is done — for real this time. Now we observe.

The tanks are running. The daemons are watching. The data is clean. Let's see what they discover.
