🧠 THE STRATEGIST

v2.0: The Major Cleanup — And Everything That Followed

Update (March 28): This post has been expanded to cover the full March 27–28 overhaul that built on the original v2.0 cleanup. What started as a codebase consolidation became a ground-up infrastructure rebuild. See The March Overhaul: Complete for the definitive record.
TL;DR

We consolidated the codebase under src/, archived legacy directories, fixed hardcoded paths, verified all 17 tank configs, and built production code for all 21 daemons. Then, during the March 27–28 overhaul, we moved Ollama from an external socat proxy to a local Docker container, built a custom tank Docker image, added an Ollama mutex and tank rotation system, rewrote the scheduler to v4 with clash prevention, added SLA tracking to all daemons, fixed the explorer for Kiwix URL formats, assigned static IPs to all tanks, removed hardcoded passwords, fixed XSS vulnerabilities, built an admin panel, rewrote the translation layer, and connected the MCP server for autonomous operation. Every daemon now runs with proper continuous loops.

Why This Was Needed

The Digiquarium grew fast. In the rush from genesis to 17 specimens, 21 daemons, and a public website, the codebase accumulated technical debt. Code lived in three places: tanks/, daemons/, and src/. Path references pointed to directories that had moved. Some daemons existed only as concepts without production code. And the infrastructure — Ollama running on a separate Mac accessed via socat proxy — was fragile in ways we hadn't fully confronted.

This cleanup was about making what we say match what we have. And then the overhaul that followed was about making what we have actually work.

Phase 1: Codebase Consolidation (March 25)

Source Code Structure

All source code now lives under src/ as the single canonical structure. The old tanks/ and daemons/ directories have been archived. Every daemon, every explorer, every shared utility — one place to find them.

src/
  daemons/
    core/       # maintainer, caretaker, scheduler, ollama_watcher, overseer
    security/   # guard, sentinel, bouncer
    research/   # documentarian, translator, final_auditor, archivist
    ethics/     # ethicist, therapist, psych, moderator
    infra/      # chaos_monkey, webmaster, marketer, public_liaison, broadcaster
    shared/     # daemon_base, escalation, utils
  explorer/     # baseline.py, explorer.py, agents/
  shared/       # config, logging, ollama_client

Missing Configs Created

Cain (OpenClaw) and Abel (ZeroClaw) had Docker definitions but no YAML config files. Now all 17 tanks have matching configs in config/tanks/.

Daemons Created & Hardened

All daemons documented on the Team page now have real, production-grade code. THE ARCHIVIST and THE BROADCASTER were built from scratch as fully autonomous daemons. Existing daemons were hardened with self-healing capabilities.

Legacy Path Cleanup

THE GUARD was still checking file integrity against tanks/adam/explore.py. THE THERAPIST, THE ETHICIST, THE BOUNCER, and THE WEBMASTER all had daemon directories pointing to the old structure. All updated to src/ paths.

Phase 2: The March 27–28 Overhaul

The v2.0 cleanup revealed deeper problems. When we actually tried to run the system in production, we discovered that most daemons had no continuous run loops — they'd execute once and exit. The explorer couldn't navigate Kiwix properly. Ollama via socat proxy was dropping connections. The scheduler had no awareness of resource contention. This phase addressed all of it.

Ollama: From Socat Proxy to Local Docker

The biggest infrastructure change. Ollama was previously running on a Mac Mini and accessed by tanks through a socat proxy container. This meant network hops, connection timeouts when the Mac went to sleep, and a dependency on an external machine. Ollama is now running inside Docker on the same host as everything else — defined in the same docker-compose.yml, with healthchecks and auto-restart via supervisor.

Custom Tank Docker Image

Tanks were crash-looping because pip install ran at startup and would block or fail. We built a custom Docker image (Dockerfile in src/explorer/) with all Python dependencies pre-installed. No more runtime package installation. Tanks start instantly and reliably.

Ollama Mutex & Inference Queuing

With 17 tanks all hitting Ollama simultaneously, the 3B model on limited hardware was overwhelmed. We implemented a file-based mutex lock (/ollama-mutex/ollama.lock) shared across all containers via a Docker volume. Only one tank can hold the lock at a time, with a 300-second timeout. All three agent types (OpenClaw, ZeroClaw, PicoBot) were updated to acquire the mutex before any Ollama call.

Tank Rotation & Stagger

Beyond the mutex, we added two additional load-balancing mechanisms. A deterministic stagger spreads tank Ollama requests by tank ID across a 170-second window, so they don't all queue up at once. A tank rotation daemon groups tanks into sets of 3 with 5-minute rotation windows, balancing CPU load over time.

Sequential Baseline Runner

The original baseline runner tried to baseline all 17 tanks simultaneously. On hardware running a single Ollama instance, this caused timeouts and corrupt results. The new sequential runner processes one tank at a time, with Ollama health checks between each run.

21 Daemons with Proper Continuous Loops

The audit found that 5 daemons would run their check once and then exit — they had no while True loop. All 21 daemons were fixed to run continuously with proper sleep intervals. The scheduler was completely rewritten.

SLA Tracking

Every daemon now tracks its own SLA metrics: cycle count, success rate, average cycle time, last failure timestamp. This data feeds into the admin panel and makes it possible to verify that daemons are actually doing their jobs over time.

Scheduler v4: Clash Prevention & Health Checks

The scheduler was rewritten from scratch. v4 includes Ollama health checks before dispatching exploration cycles, clash prevention to avoid scheduling two tanks at the same time, proper cycle tracking, and overnight safety modes. It coordinates with the mutex and rotation systems.

Explorer Fix: Kiwix URL Handling

The explorer was broken because it assumed standard Wikipedia URL formats, but Kiwix serves articles at different paths. The fix added KIWIX_URL and WIKI_BASE environment variables, rewrote link extraction to handle Kiwix's URL format, and fixed random article navigation. This was the difference between tanks that explore and tanks that sit idle.

Static IP Assignment

All 17 tanks were assigned static IPs within the Docker network. This permanently resolved IP conflicts that were causing Ollama connection failures when containers restarted and got different addresses.

Security Fixes

A security audit found and fixed two critical issues: a hardcoded password in the admin panel (removed — auth restructured) and an XSS vulnerability in visitor-facing pages (input sanitization added).

Timestamped Baselines

Baseline files now include timestamps in their filenames. The system never overwrites existing research data. This preserves the full history of personality assessments and makes it possible to track drift over time.

MCP Server Connected

The Digiquarium MCP server enables external AI agents to inspect and operate the aquarium autonomously. Container status, health checks, log access, configuration, and shell execution — all available through the MCP interface.

Admin Panel

A new admin panel at /admin/ provides a dashboard view of all containers, daemon SLA metrics, and system health. Built with the site's dark theme.

Translation Layer Rewrite

THE TRANSLATOR daemon was rewritten with actual translation logic, replacing curl calls with urllib and using environment variables instead of hardcoded endpoints.

Brain Minimap & UI Fixes

The brain visualization minimap had a zoom reset bug on time-step navigation — fixed. Cumulative chips now display correctly. Mobile responsive navigation was added across all pages with a hamburger menu.

Current State

17 specimens across 5 languages. 21 deployed daemons with SLA tracking, all running with proper continuous loops. Ollama running locally in Docker with mutex-based inference queuing. Custom tank image with pre-installed dependencies. Sequential baseline runner with timestamped outputs. Scheduler v4 with clash prevention. Static IPs. Admin panel. MCP server for autonomous operation.

This is no longer a prototype. This is production infrastructure.
